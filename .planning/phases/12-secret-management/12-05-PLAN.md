<task type="auto">
  <name>Create deployment secrets documentation</name>
  <files>docs/DEPLOYMENT_SECRETS.md</files>
  <action>
    Create docs/DEPLOYMENT_SECRETS.md with:

    1. Overview section explaining:
       - What deployment secrets are (vs tenant credentials from Phase 1)
       - Why they're separate (deployment-time vs runtime)
       - Two-tier architecture: GitHub Environments + SSM Parameter Store

    2. SSM Parameter Hierarchy:
       ```
       /viberator/{environment}/{category}/{key}

       Categories:
       - database: url, host (SecureString)
       - frontend: apiUrl (String)
       - amplify: appId, branch (String)
       - ecs: cluster, service (String)
       - deployment: oidcRoleArn, ecrRepository, region (SecureString for role)
       ```

    3. GitHub Environment Setup:
       - Create environments: dev, staging, prod
       - Add secrets per environment:
         * AWS_ROLE_ARN (OIDC role for that environment)
         * AMPLIFY_APP_ID (Amplify app for that environment)
         * AMPLIFY_BRANCH (branch name, optional, defaults to main)

    4. Initial SSM Setup (CLI commands):
       ```bash
       # Database (SecureString - encrypted)
       aws ssm put-parameter \
         --name "/viberator/dev/database/url" \
         --value "postgresql://..." \
         --type "SecureString" \
         --overwrite

       # Frontend API URL (String - not sensitive)
       aws ssm put-parameter \
         --name "/viberator/dev/frontend/apiUrl" \
         --value "https://dev-api.viberator.com" \
         --type "String" \
         --overwrite

       # Amplify (String - not sensitive)
       aws ssm put-parameter \
         --name "/viberator/dev/amplify/appId" \
         --value "d1234567890abc" \
         --type "String" \
         --overwrite
       ```

    5. Pulumi Integration:
       - How createDeploymentSecrets() provisions parameters
       - Setting secret values via pulumi config (with --secret flag)
       - KMS key configuration for SecureString encryption

    6. Workflow Usage:
       - How workflows fetch from SSM during deployment
       - Environment: directive provides secret namespace
       - OIDC authentication eliminates long-lived credentials

    7. Troubleshooting section:
       - "Parameter doesn't exist" -> Run Pulumi up to create parameters
       - "Access denied" -> Check IAM role has ssm:GetParameter permission
       - "Wrong value in production" -> Check using correct environment in SSM path
       - "Secrets not updating in ECS" -> Force new deployment after SSM change

    8. Security Best Practices:
       - Use SecureString for sensitive values
       - Rotate OIDC role credentials (no rotation needed - it's a role ARN)
       - Never log secret values
       - Use KMS customer-managed key for encryption
  </action>
