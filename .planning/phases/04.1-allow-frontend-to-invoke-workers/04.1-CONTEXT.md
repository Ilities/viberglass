# Phase 4.1: Allow Frontend to Invoke Workers - Context

**Gathered:** 2026-01-20
**Status:** Ready for planning

<domain>
## Phase Boundary

Frontend can invoke workers and initiate jobs from tickets. This phase bridges the UI to the worker execution layer built in Phase 4. Users trigger job runs from tickets, see confirmation with details, and land on a job detail page.

</domain>

<decisions>
## Implementation Decisions

### Trigger Point
- Run action available in **both** ticket list view and ticket detail view
- List view: **icon button** (play icon) in each row
- Detail view: button in the **actions section** (alongside edit, delete, etc.)
- Button **shows target clanker** name (not generic "Run")
- If project has multiple clankers: uses **project default**, user can change in pre-flight modal
- **Single ticket at a time** — no bulk running for now
- Visibility (always visible vs hover): Claude's discretion based on existing UI patterns

### Pre-flight Modal
- **Confirmation modal** required before running
- Modal shows **detailed info**: ticket title, description, clanker config, estimated cost/time
- **Dropdown** to select different clanker if multiple available
- Ticket info displayed **read-only** in modal (inline editing deferred - see below)

### Feedback Flow
- After confirm: **toast notification** with "View job" link + **immediate redirect** to job detail page
- Confirm button shows **spinner** while API call in progress
- **Job detail page needs creation** in this phase
- Job detail page shows **full status**: job ID, status, logs, PR link, error details when available
- **No auto-polling** — page is static for now, Phase 5 adds polling

### Edge Cases
- No clanker configured: clicking run **opens clanker setup** prompt
- Job already running for ticket: **warn** but allow if user confirms
- Ticket missing required fields: **button disabled**, tooltip shows what's missing
- Required fields: Claude discovers from existing ticket model

### Claude's Discretion
- Icon button visibility in list (always vs hover)
- UI component library/style system (discover from codebase)
- Specific loading state implementation
- Toast duration and styling

</decisions>

<specifics>
## Specific Ideas

- Button should indicate which clanker will be used, not just generic "Run"
- Modal is detailed — users see what they're about to execute before confirming
- Immediate redirect after confirmation — user lands on job detail page to see progress
- Toast has link so user can navigate back to job if they were redirected elsewhere

</specifics>

<deferred>
## Deferred Ideas

- Bulk running multiple tickets — considered but explicitly out of scope
- Auto-polling on job detail page — Phase 5 handles this
- **Inline ticket editing in modal** — originally discussed but deferred to keep phase scope manageable. Users can edit ticket on detail page before clicking run. This keeps the modal focused on confirmation and clanker selection.

</deferred>
