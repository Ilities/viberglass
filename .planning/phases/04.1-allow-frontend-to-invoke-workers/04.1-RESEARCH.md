# Phase 4.1: Allow Frontend to Invoke Workers - Research

**Researched:** 2026-01-20
**Domain:** Frontend-to-Backend Worker Invocation Flow
**Confidence:** HIGH

## Summary

This phase bridges the frontend UI to the existing worker execution infrastructure (built in Phase 4). The research reveals that all backend components are in place: `WorkerExecutionService`, `WorkerInvoker` implementations (Lambda, ECS, Docker), and `JobService`. The frontend has a component library with Dialog, Button, Select/Listbox, and a pattern for client-side actions (`ClankerActions.tsx`).

The primary gap is the **API endpoint** to create a job from a ticket and invoke a worker, plus the **frontend components** for the run flow (modal, list actions, job detail page). A secondary gap is the **project-clanker relationship** - projects don't currently have default clankers, but the decision from CONTEXT.md allows using a dropdown in the pre-flight modal.

**Primary recommendation:** Create a new `POST /api/tickets/:ticketId/run` endpoint that creates a job from ticket data, resolves the clanker, and invokes `WorkerExecutionService.executeJob()`. Frontend adds run button, pre-flight modal, and job detail page.

**When to use:** When frontend needs to trigger a complete flow
**Example:**

### Pattern 2: Pre-flight Modal Pattern
**What:** Confirmation dialog with editable details before action
**When to use:** For destructive or expensive operations
**Example:**
```typescript
// Source: Existing pattern in clanker-actions.tsx
function RunTicketModal({ ticket, clankers, open, onClose }) {
  const [selectedClanker, setSelectedClanker] = useState(clankers[0])
  const [isSubmitting, setIsSubmitting] = useState(false)

  async function handleRun() {
    setIsSubmitting(true)
    try {
      const job = await runTicket(ticket.id, selectedClanker.id)
      router.push(`/project/${project}/jobs/${job.jobId}`)
    } finally {
      setIsSubmitting(false)
    }
  }

  return (
    <Dialog open={open} onClose={onClose}>
      <DialogTitle>Run Ticket</DialogTitle>
      <DialogBody>
        {/* Ticket details, clanker selector */}
      </DialogBody>
      <DialogActions>
        <Button plain onClick={onClose}>Cancel</Button>
        <Button color="brand" disabled={isSubmitting} onClick={handleRun}>
          {isSubmitting ? 'Starting...' : 'Run'}
        </Button>
      </DialogActions>
    </Dialog>
  )
}
```

### Anti-Patterns to Avoid
- **Direct WorkerInvoker calls from API routes:** Use WorkerExecutionService which handles retry logic
- **Synchronous job execution in API:** Worker invocation is async, return jobId immediately
- **Creating job without invoking worker:** Always invoke in same transaction for consistency
- **Polling in this phase:** CONTEXT.md explicitly defers polling to Phase 5

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Job-to-worker invocation | Custom invoke logic | WorkerExecutionService | Has retry logic, error handling, status updates |
| Clanker resolution | Inline clanker lookup | ClankerDAO.getClanker() | Handles join with deployment strategy |
| Modal dialogs | Custom modal | Dialog from @headlessui | Accessible, animated, keyboard nav |
| Dropdown selection | Custom dropdown | Listbox or Select | Accessible, styled consistently |
| Loading states | Custom spinner | Button with disabled + text change | Pattern in clanker-actions.tsx |
| Toast notifications | console.log or alert | sonner | Non-blocking, auto-dismiss, links |

**Key insight:** All UI primitives exist. The work is composing them into new components, not building components from scratch.

## Common Pitfalls

### Pitfall 1: Missing Tenant ID
**What goes wrong:** Job created without tenant_id, callback endpoint rejects result
**Why it happens:** Ticket doesn't directly have tenantId, must derive from project
**How to avoid:** Get tenantId from project context or use default "api-server" if not multi-tenant
**Warning signs:** Jobs show as "queued" forever, callback returns 403

### Pitfall 2: No Clanker Available
**What goes wrong:** User clicks run but no clankers exist
**Why it happens:** Clankers are global, not project-scoped
**How to avoid:**
1. Check clanker availability before showing run button
2. Show "No clankers configured" prompt (per CONTEXT.md decision)
**Warning signs:** Empty dropdown in modal, 500 errors on run

### Pitfall 3: Ticket Missing Repository
**What goes wrong:** Job created but worker fails - no repository to work with
**Why it happens:** Tickets don't have repository, projects do
**How to avoid:** Get repository from project.repositoryUrl
**Warning signs:** Worker returns error "No repository specified"

### Pitfall 4: Stale Job Status on Detail Page
**What goes wrong:** User lands on job page, sees "queued" but job is running
**Why it happens:** Initial render shows DB state, worker hasn't updated yet
**How to avoid:**
1. Accept this for Phase 4.1 (no polling - CONTEXT.md decision)
2. Show "Status may be outdated" hint
3. Add manual refresh button
**Warning signs:** Users confused about job state

### Pitfall 5: Race Between Modal Close and Navigate
**What goes wrong:** User sees flash of previous page before redirect
**Why it happens:** Modal animates close while router navigates
**How to avoid:** Navigate first, don't wait for modal close animation
**Warning signs:** Visual glitch on successful submission

```typescript
// Source: Derived from JobService.submitJob() and WorkerExecutionService.executeJob()
async function runTicketAsJob(
  ticket: Ticket,
  project: ProjectConfig,
  clanker: Clanker
): Promise<{ jobId: string; executionId?: string }> {
  const jobId = `job_${Date.now()}_${Math.random().toString(36).substr(2, 9)}`;

  const jobData: JobData = {
    id: jobId,
    tenantId: 'api-server', // Or derive from project/user
    repository: project.repositoryUrl || '',
    task: `${ticket.title}\n\n${ticket.description}`,
    branch: undefined, // Worker creates branch
    baseBranch: 'main',
    context: {
      stepsToReproduce: ticket.description,
      consoleErrors: ticket.metadata.errors?.map(e => e.message),
      affectedFiles: [], // Could derive from metadata
    },
    settings: {
      testRequired: false,
      maxChanges: 10,
    },
    timestamp: Date.now(),
  };

  // Create job record
  await jobService.submitJob(jobData);

  // Invoke worker (handles retries internally)
  const result = await workerExecutionService.executeJob(jobData, clanker);

  return {
    jobId,
    executionId: result.executionId,
  };
}
```

### Dialog with Clanker Selection (Frontend)
```typescript
// Source: Pattern from Dialog component + Listbox
import { Dialog, DialogTitle, DialogBody, DialogActions } from '@/components/dialog'
import { Listbox, ListboxOption, ListboxLabel } from '@/components/listbox'
import { Button } from '@/components/button'

interface RunTicketModalProps {
  ticket: Ticket | null
  clankers: Clanker[]
  open: boolean
  onClose: () => void
  onRun: (ticketId: string, clankerId: string) => Promise<void>
}

export function RunTicketModal({ ticket, clankers, open, onClose, onRun }: RunTicketModalProps) {
  const [selectedClanker, setSelectedClanker] = useState<Clanker | null>(clankers[0] || null)
  const [isRunning, setIsRunning] = useState(false)

  async function handleRun() {
    if (!ticket || !selectedClanker) return
    setIsRunning(true)
    try {
      await onRun(ticket.id, selectedClanker.id)
      // Navigation happens in parent after success
    } finally {
      setIsRunning(false)
    }
  }

  return (
    <Dialog open={open} onClose={onClose} size="lg">
      <DialogTitle>Run Ticket with Clanker</DialogTitle>
      <DialogBody>
        {ticket && (
          <div className="space-y-4">
            <div>
              <h4 className="font-medium">Ticket</h4>
              <p className="text-zinc-600 dark:text-zinc-400">{ticket.title}</p>
            </div>
            <div>
              <h4 className="font-medium mb-2">Select Clanker</h4>
              <Listbox value={selectedClanker} onChange={setSelectedClanker}>
                {clankers.map(clanker => (
                  <ListboxOption key={clanker.id} value={clanker}>
                    <ListboxLabel>{clanker.name}</ListboxLabel>
                  </ListboxOption>
                ))}
              </Listbox>
            </div>
          </div>
        )}
      </DialogBody>
      <DialogActions>
        <Button plain onClick={onClose} disabled={isRunning}>Cancel</Button>
        <Button
          color="brand"
          disabled={isRunning || !selectedClanker}
          onClick={handleRun}
        >
          {isRunning ? 'Starting...' : `Run with ${selectedClanker?.name || 'Clanker'}`}
        </Button>
      </DialogActions>
    </Dialog>
  )
}
```

## Data Model Observations

### Current State
- **Tickets** have `projectId` (link to project) and `autoFixStatus` (pending/in_progress/completed/failed)
- **Jobs** have `tenant_id`, `repository`, `task` - no direct ticket reference
- **Clankers** are global, not project-scoped
- **Projects** have `repositoryUrl` but no `defaultClankerId`

### Implications for Implementation
1. **Ticket-to-Job mapping:** Must store reference somehow (options: job.context, new field, or ticket.lastJobId)
2. **Clanker selection:** List all clankers, let user pick (no project default)
3. **Repository source:** Get from `project.repositoryUrl`
4. **Task text:** Compose from `ticket.title` + `ticket.description`

### Recommended Schema Addition
Consider adding to jobs table:
```sql
ticket_id UUID REFERENCES tickets(id) NULL,
clanker_id UUID REFERENCES clankers(id) NULL,
```
This enables:
- Linking job back to originating ticket
- Tracking which clanker executed the job
- Future: showing job history on ticket detail page

## Required Fields Check

Based on CONTEXT.md: "Ticket missing required fields: button disabled, tooltip shows what's missing"

**Ticket required fields for running:**
1. `title` - Required (always present, has min validation)
2. `description` - Required (always present)
3. `projectId` - Required (foreign key)

**Project required fields for running:**
1. `repositoryUrl` - Must have a target repository

**Clanker required fields for running:**
1. Must have at least one clanker in the system
2. Selected clanker must have `deploymentStrategyId` and `deploymentConfig`

**Validation logic:**
```typescript
function canRunTicket(ticket: Ticket, project: Project, clankers: Clanker[]): { canRun: boolean; reason?: string } {
  if (!project.repositoryUrl) {
    return { canRun: false, reason: 'Project has no repository configured' }
  }
  if (clankers.length === 0) {
    return { canRun: false, reason: 'No clankers available' }
  }
  const validClankers = clankers.filter(c => c.deploymentStrategyId && c.status === 'active')
  if (validClankers.length === 0) {
    return { canRun: false, reason: 'No active clankers with deployment configured' }
  }
  return { canRun: true }
}
```

## Open Questions

Things that couldn't be fully resolved:

1. **Job-Ticket Reference Storage**
   - What we know: Jobs don't currently reference tickets
   - What's unclear: Should we add `ticket_id` to jobs table or store in `context`?
   - Recommendation: Add `ticket_id` column for proper relational integrity

2. **Multiple Jobs per Ticket**
   - What we know: CONTEXT.md says "warn but allow if user confirms"
   - What's unclear: How to detect "job already running"?
   - Recommendation: Query jobs where ticket_id matches and status is 'queued' or 'active'

3. **Tenant ID Source**
   - What we know: Jobs require tenant_id, current code defaults to 'api-server'
   - What's unclear: Is there a user/org context to derive real tenant?
   - Recommendation: Use 'api-server' for now, refine when auth is implemented
