  <files>viberator/app/src/workers/types.ts</files>
  <action>
    Create payload interfaces in viberator/app/src/workers/types.ts:

    1. Create BaseWorkerPayload interface with shared fields:
       - tenantId: string
       - jobId: string
       - clankerId: string
       - repository: string
       - task: string
       - branch?: string
       - baseBranch?: string
       - context?: { stepsToReproduce, expectedBehavior, actualBehavior, stackTrace, consoleErrors, affectedFiles }
       - settings?: { maxChanges, testRequired, codingStandards, runTests, testCommand, maxExecutionTime }

    2. Create S3InstructionFile interface:
       - fileType: string (e.g., 'agents.md', 'claude.md')
       - s3Url: string (s3://bucket/key format)

    3. Create MountedInstructionFile interface:
       - fileType: string
       - mountPath: string (e.g., /etc/viberator/config/agents.md)

    4. Create LambdaPayload extending BaseWorkerPayload:
       - workerType: 'lambda'
       - instructionFiles: S3InstructionFile[]
       - requiredCredentials: string[] (e.g., ['GITHUB_TOKEN', 'CLAUDE_API_KEY'])
       - deploymentConfig?: Record<string, unknown>

    5. Create EcsPayload extending BaseWorkerPayload (same structure as Lambda):
       - workerType: 'ecs'
       - instructionFiles: S3InstructionFile[]
       - requiredCredentials: string[]
       - deploymentConfig?: Record<string, unknown>

    6. Create DockerPayload extending BaseWorkerPayload:
       - workerType: 'docker'
       - instructionFiles: MountedInstructionFile[] (different from AWS - uses mount paths)
       - requiredCredentials: string[]
       - clankerConfig?: Record<string, unknown> (full config for Docker)

    7. Create WorkerPayload union type:
       export type WorkerPayload = LambdaPayload | EcsPayload | DockerPayload

    Keep existing CodingJobData and JobResult interfaces - new payload types are additive, not replacements.
