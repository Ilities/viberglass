# Phase 4.4: E2E Flow Verification - Research

**Researched:** 2026-01-21
**Domain:** End-to-end testing, infrastructure documentation, Docker/ECS verification
**Confidence:** HIGH

## Summary

This phase is a **verification and documentation phase**, not a development phase. The goal is to verify the existing implementation works end-to-end and document the infrastructure setup for both local Docker and AWS ECS deployments.

The codebase already has all components implemented:
- Frontend: ticket creation, clanker selection, job triggering, job status display
- Backend: ticket/job APIs, worker execution service, callback handling
- Worker: ViberatorWorker with Docker/ECS/Lambda support, CallbackClient for results

The key research finding is that **log streaming to frontend is NOT currently implemented** - logs are collected at job completion and stored, but not streamed in real-time. This is a gap that needs to be documented (Phase 7 covers runtime status/progress updates via STAT-02/STAT-03/STAT-04/STAT-05).

**Primary recommendation:** Focus on manual verification with structured checklists, identify gaps, and produce actionable infrastructure documentation - no new development work needed.

## Common Pitfalls

### Pitfall 1: Skipping Prerequisites
**What goes wrong:** Verification fails because environment not properly configured
**Why it happens:** Assumptions about existing setup
**How to avoid:** Document explicit prerequisites, verify each before starting
**Warning signs:** "Works on my machine" failures

### Pitfall 2: Testing Without Database
**What goes wrong:** Jobs created but never stored, callback fails
**Why it happens:** Backend running but DB not started
**How to avoid:** Verify DB connection before any tests
**Verification:** `curl http://localhost:8888/api/jobs/stats/queue` returns valid JSON

### Pitfall 3: Docker Socket Permissions
**What goes wrong:** DockerInvoker cannot communicate with Docker daemon
**Why it happens:** Backend process lacks socket access
**How to avoid:** Document socket path and permissions requirements
**Warning signs:** ECONNREFUSED or EACCES errors

### Pitfall 4: Missing GitHub Credentials
**What goes wrong:** Worker clones repo but cannot push branch or create PR
**Why it happens:** GITHUB_TOKEN not set or lacks permissions
**How to avoid:** Document credential requirements with scope details
**Verification:** Token has `repo` scope for pushing

### Pitfall 5: Callback URL Mismatch
**What goes wrong:** Worker executes successfully but platform never gets result
**Why it happens:** PLATFORM_API_URL wrong or not accessible from container
**How to avoid:** Network configuration documentation
**Verification:** Container can curl platform API

### Pitfall 6: Clanker Not Active
**What goes wrong:** Run button fails with "Clanker not ready"
**Why it happens:** Clanker status not 'active' or no deploymentStrategyId
**How to avoid:** Document clanker setup steps
**Verification:** API returns clanker with status='active' and deploymentStrategyId set

### Future: SSE Log Streaming (Phase 7 Reference)
```typescript
// Reference: https://dev.to/manojspace/real-time-log-streaming-with-nodejs-and-react-using-server-sent-events-sse-48pk
// Backend endpoint pattern
app.get('/api/jobs/:jobId/logs/stream', (req, res) => {
  res.setHeader('Content-Type', 'text/event-stream');
  res.setHeader('Cache-Control', 'no-cache');
  res.setHeader('Connection', 'keep-alive');

  // Send logs as they arrive
  const sendLog = (line: string) => {
    res.write(`data: ${JSON.stringify({ log: line })}\n\n`);
  };

  // Cleanup on client disconnect
  req.on('close', () => { /* cleanup */ });
});

// Frontend
const eventSource = new EventSource(`/api/jobs/${jobId}/logs/stream`);
eventSource.onmessage = (event) => {
  const { log } = JSON.parse(event.data);
  appendLog(log);
};
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| WebSocket for logs | SSE (Server-Sent Events) | Standard since HTML5 | Simpler, built-in reconnection |
| Custom polling | React Query / SWR | 2023+ | Automatic refetching, caching |
| Manual E2E tests | Playwright | 2022+ | Reliable browser automation |

**Current implementation:**
- Job status: Manual refresh button (adequate for verification phase)
- Logs: Collected at completion, not streamed (Phase 7 scope)

**Deprecated/outdated:**
- Long-polling: Use SSE instead for server-to-client streaming
- Socket.io: Overkill for unidirectional log streaming, SSE is simpler

## Identified Gaps

Based on codebase analysis:

### Gap 1: Log Streaming (Phase 7 Scope)
**Current:** `logs: []` passed to callback, no collection mechanism
**Expected:** Real-time log streaming during job execution
**Code location:** `viberator/app/src/workers/viberator.ts:230`
**Future work:** Phase 7 (STAT-03: Worker POSTs progress updates)

### Gap 2: Job Progress Updates (Phase 7 Scope)
**Current:** Job page shows static status, manual refresh required
**Expected:** Auto-updating status during execution
**Code location:** `platform/frontend/src/app/(app)/project/[project]/jobs/[jobId]/page.tsx`
**Future work:** Phase 5 (CB-04: Frontend polls job status) and Phase 7

### Gap 3: ECS Log Viewing
**Current:** Logs go to CloudWatch, not surfaced in platform

**Future work:** CloudWatch Logs API integration (Phase 7)
