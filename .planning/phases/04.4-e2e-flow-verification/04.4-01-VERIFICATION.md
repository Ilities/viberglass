## Code Flow Trace

### 1. Frontend -> API

**Entry Point:** `platform/frontend/src/components/run-ticket-modal.tsx`

**Flow:**
1. User clicks "Run" button on a ticket
2. Modal opens showing available clankers
3. Clankers filtered by: `c.status === 'active' && c.deploymentStrategyId` (line 22)
4. User selects clanker from dropdown and confirms
5. `handleRun()` calls `runTicket(ticket.id, selectedClanker.id)` (line 34)
6. `runTicket()` in `src/service/api/job-api.ts` makes:
   - `POST /api/tickets/${ticketId}/run`
   - Body: `{ clankerId }`
7. On success, toast notification shown with "View Job" action
8. Router navigates to `/project/${project}/jobs/${jobId}`

**Key files:**
- `platform/frontend/src/components/run-ticket-modal.tsx` - Modal component (lines 20-115)
- `platform/frontend/src/service/api/job-api.ts` - API client (lines 48-63)

**Status:** IMPLEMENTED

**Gap:** None identified for this stage

### 2. API -> Worker Invocation

**Entry Point:** `platform/backend/src/api/routes/tickets.ts:270`

**Flow:**
1. `POST /:id/run` receives request
2. Validates ticket exists (via `ticketService.getTicket`)
3. Validates project exists and has `repositoryUrl`
4. Validates clanker exists, has `deploymentStrategyId`, and `status === 'active'`
5. Generates jobId: `job_${Date.now()}_${randomUUID().slice(0, 8)}`
6. Creates job via `jobService.submitJob(jobData, { ticketId, clankerId })`
7. Invokes worker via `workerExecutionService.executeJob(jobData, clanker)`
8. Returns `202 Accepted` immediately (fire-and-forget)

**Key files:**
- `tickets.ts:270-396` - Run ticket endpoint
- `JobService.ts` - Job creation and storage
- `WorkerExecutionService.ts` - Worker invocation orchestration

**Status:** IMPLEMENTED

**Gap:** None identified for this stage

---

### 3. Worker Invocation -> Docker Container

**Entry Point:** `platform/backend/src/workers/WorkerExecutionService.ts`

**Flow:**
1. `executeJob()` receives job and clanker
2. `WorkerInvokerFactory.create(clanker)` returns appropriate invoker
3. For Docker clankers: Returns `DockerInvoker`
4. `DockerInvoker.invoke()` called
5. Reads deployment config from `clanker.deploymentConfig.containerImage`
6. Creates container with:
   - Name: `viberator-job-${job.id}`
   - Env vars: `JOB_PAYLOAD` (JSON), `TENANT_ID`, `JOB_ID`
   - AutoRemove: true
   - NetworkMode: bridge (default)
7. `container.start()` launches the worker
8. Returns `{ executionId: containerId, workerType: "docker" }`

**Key files:**
- `WorkerExecutionService.ts` - Orchestration
- `WorkerInvokerFactory.ts` - Invoker selection
- `DockerInvoker.ts:28-83` - Docker container creation

**Status:** IMPLEMENTED

**Gap:** None identified for this stage

---

### 4. Worker Execution (Inside Container)

**Entry Point:** `viberator/app/src/workers/viberator.ts` (CLI entry)

**Flow:**
1. Container starts with `JOB_PAYLOAD` environment variable
2. `ViberatorWorker.initialize(payload)` called
3. Reads config from payload (tenantId, repository, task, etc.)
4. Fetches credentials via `CredentialProvider` (from SSM for AWS, environment for Docker)
5. Loads instruction files (from filesystem for Docker)
6. `executeTask()` called:
   - Clones repository to `/tmp/viberator-work/${jobId}/repo`
   - Creates feature branch: `fix/${jobId}`
   - Executes agent (Claude Code by default)
   - Commits changes
   - Pushes branch to GitHub
   - Creates pull request
7. Sends result to platform via `CallbackClient.sendResult()`
8. Exits (container auto-removed)

**Key files:**
- `cli-worker.ts` - CLI entry point
- `viberator.ts:134-279` - Main execution logic
- `GitService.ts` - Git operations
- `AgentOrchestrator.ts` - Agent execution

**Status:** IMPLEMENTED

**Gaps:**
1. **Log collection:** `logs: []` passed to callback (TODO comment at line 230)
2. **Credential injection:** Relies on environment variables being present in container

---

### 5. Result Callback -> Platform

**Entry Point:** `platform/backend/src/api/routes/jobs.ts:127`

**Flow:**
1. Worker POSTs to `/api/jobs/:jobId/result`
2. `tenantMiddleware` validates `X-Tenant-Id` header (SEC-03)
3. `validateResultCallback` validates payload structure
4. Job lookup verifies:
   - Job exists
   - Job belongs to tenant (tenantId match)
5. Idempotency check: Rejects if job already in `completed` or `failed` state (409 Conflict)
6. Status determined from `result.success` boolean
7. `jobService.updateJobStatus()` stores:
   - `result.success`, `result.branch`, `result.pullRequestUrl`
   - `result.changedFiles`, `result.executionTime`, `result.errorMessage`
   - `result.commitHash`
8. Returns 200 OK with `{ success: true, jobId, status }`

**Key files:**
- `platform/backend/src/api/routes/jobs.ts:127-186` - Result callback endpoint
- `viberator/app/src/workers/CallbackClient.ts` - Worker-side HTTP client

**Status:** IMPLEMENTED

**Gap:** Logs are currently empty array (`logs: []`) - not collected during execution (viberator.ts:230, viberator.ts:258)
