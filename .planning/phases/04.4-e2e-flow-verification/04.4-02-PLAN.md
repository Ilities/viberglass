## Overview
- What this doc covers (local development with Docker clanker)
- Architecture diagram showing: Frontend -> Backend -> DockerInvoker -> Container -> Callback

## Prerequisites
- Docker Desktop or Docker Engine installed
- Node.js 18+
- PostgreSQL (can use docker-compose)
- GitHub Personal Access Token with `repo` scope
- Anthropic API key (for Claude agent)

## Step 1: Build Worker Image
```bash
cd viberator/app
npm install
npm run build
npm run docker:build:local  # or docker build -t viberator-worker:local .
```

Verify: `docker images | grep viberator-worker`

## Step 2: Start Platform Services
```bash
# Start PostgreSQL (if not running)
cd platform/backend
docker-compose up -d postgres

# Apply migrations
npm run db:migrate

# Start backend
npm run dev
```

Verify: `curl http://localhost:8888/health`

## Step 3: Start Frontend
```bash
cd platform/frontend
npm run dev
```

Verify: Open http://localhost:3000

## Step 4: Configure Clanker
1. Navigate to Clankers page in UI
2. Create new clanker with:
   - Name: "Local Docker"
   - Deployment Strategy: Select Docker strategy
   - Status: Active
   - Deployment Config:
     ```json
     {
       "containerImage": "viberator-worker:local",
       "networkMode": "host",
       "environmentVariables": {
         "PLATFORM_API_URL": "http://host.docker.internal:8888",
         "GITHUB_TOKEN": "ghp_xxx",
         "ANTHROPIC_API_KEY": "sk-ant-xxx"
       }
     }
     ```

Note on Linux: Replace `host.docker.internal` with `172.17.0.1` (Docker bridge IP)

## Step 5: Create and Run Ticket
1. Create a project with a repository URL
2. Create a ticket describing a bug or feature
3. Click "Run" and select the Docker clanker
4. Watch job status (manual refresh for now)

## Troubleshooting

### Docker Socket Permission Denied
Error: `ECONNREFUSED` or `EACCES` when starting container
Solution: Add user to docker group: `sudo usermod -aG docker $USER`

### Container Cannot Reach Platform API
Error: Callback fails with connection refused
Solution:
- Linux: Use `172.17.0.1` instead of `host.docker.internal`
- Mac/Windows: Verify `host.docker.internal` resolves inside container

### Image Not Found
Error: `no such image: viberator-worker:local`
Solution: Run `npm run docker:build:local` in viberator/app

### GitHub Push Fails
Error: `remote: Permission denied`
Solution: Ensure GITHUB_TOKEN has `repo` scope

## Known Limitations
- Log streaming not yet implemented (Phase 7)
- Manual refresh required for job status (Phase 5 adds polling)

Create comprehensive AWS ECS clanker setup documentation:

Write AWS_ECS_SETUP.md with these sections:

## Overview
- What this doc covers (production/staging AWS ECS deployment)
- Architecture: Platform -> EcsInvoker -> ECS Fargate -> Callback
- Cost considerations (Fargate pricing per task)

## Prerequisites
- AWS CLI configured with appropriate permissions
- AWS account with ECS, ECR, SSM, CloudWatch access
- VPC with public subnets (for Fargate with public IP)
- Platform deployed and accessible (e.g., on AWS ECS or EC2)
- GitHub PAT and Anthropic API key

## Required AWS Permissions
```json
{
  "Statement": [
    { "Effect": "Allow", "Action": ["ecs:RunTask", "ecs:DescribeTasks"], "Resource": "*" },
    { "Effect": "Allow", "Action": ["ssm:GetParameter", "ssm:PutParameter"], "Resource": "arn:aws:ssm:*:*:parameter/viberator/*" },
    { "Effect": "Allow", "Action": ["ecr:GetAuthorizationToken", "ecr:BatchGetImage", "ecr:GetDownloadUrlForLayer"], "Resource": "*" },
    { "Effect": "Allow", "Action": ["logs:CreateLogStream", "logs:PutLogEvents"], "Resource": "*" },
    { "Effect": "Allow", "Action": ["iam:PassRole"], "Resource": "<task-execution-role-arn>" }
  ]
}
```

## Step 1: Create ECR Repository and Push Image
```bash
# Create repository
aws ecr create-repository --repository-name viberator-worker

# Login to ECR
aws ecr get-login-password --region eu-west-1 | docker login --username AWS --password-stdin <account>.dkr.ecr.eu-west-1.amazonaws.com

# Build and push
cd viberator/app
docker build -t viberator-worker .
docker tag viberator-worker:latest <account>.dkr.ecr.eu-west-1.amazonaws.com/viberator-worker:latest
docker push <account>.dkr.ecr.eu-west-1.amazonaws.com/viberator-worker:latest
```

## Step 2: Store Credentials in SSM Parameter Store
```bash
# Store GitHub token
aws ssm put-parameter \
  --name "/viberator/tenants/api-server/GITHUB_TOKEN" \
  --value "ghp_xxx" \
  --type SecureString

# Store Anthropic API key
aws ssm put-parameter \
  --name "/viberator/tenants/api-server/ANTHROPIC_API_KEY" \
  --value "sk-ant-xxx" \
  --type SecureString
```

Verify: `aws ssm get-parameter --name "/viberator/tenants/api-server/GITHUB_TOKEN" --with-decryption`

## Step 3: Create ECS Task Definition
```json
{
  "family": "viberator-worker",
  "networkMode": "awsvpc",
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "1024",
  "memory": "2048",
  "executionRoleArn": "<execution-role-arn>",
  "taskRoleArn": "<task-role-arn>",
  "containerDefinitions": [
    {
      "name": "worker",
      "image": "<account>.dkr.ecr.eu-west-1.amazonaws.com/viberator-worker:latest",
      "essential": true,
      "logConfiguration": {
        "logDriver": "awslogs",
        "options": {
          "awslogs-group": "/ecs/viberator-worker",
          "awslogs-region": "eu-west-1",
          "awslogs-stream-prefix": "ecs"
        }
      }
    }
  ]
}
```

Register: `aws ecs register-task-definition --cli-input-json file://task-definition.json`

## Step 4: Create ECS Cluster
```bash
aws ecs create-cluster --cluster-name viberator-workers
```

## Step 5: Configure Clanker in Platform
Create clanker with deployment config:
```json
{
  "clusterArn": "arn:aws:ecs:eu-west-1:<account>:cluster/viberator-workers",
  "taskDefinitionArn": "arn:aws:ecs:eu-west-1:<account>:task-definition/viberator-worker:1",
  "subnetIds": ["subnet-xxx", "subnet-yyy"],
  "securityGroupIds": ["sg-xxx"],
  "launchType": "FARGATE",
  "assignPublicIp": "ENABLED"
}
```

## Step 6: Verify End-to-End
1. Create ticket and run with ECS clanker
2. Check ECS console for running task: `aws ecs list-tasks --cluster viberator-workers`
3. Check CloudWatch logs: `aws logs tail /ecs/viberator-worker --follow`
4. Verify PR created on GitHub

## Troubleshooting

### Task Fails to Start
Error: `STOPPED (Essential container in task exited)`
Solution: Check CloudWatch logs for container errors

### Cannot Pull Image
Error: `CannotPullContainerError`
Solution: Verify execution role has ECR permissions

### SSM Parameter Not Found
Error: `ParameterNotFound`
Solution: Ensure parameter path matches `/viberator/tenants/<tenantId>/<key>`

### Callback Fails
Error: Worker completes but platform never updates
Solution: Ensure task has internet access (public IP or NAT gateway)

## Cost Optimization
- Use Fargate Spot for development/staging (70% savings)
- Set appropriate CPU/memory for workload
- Clean up stopped tasks and unused task definitions
