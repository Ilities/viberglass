# Phase 11.2: Amplify Frontend Infrastructure - Research

**Researched:** 2026-01-23
**Domain:** AWS Amplify Infrastructure with Pulumi
**Confidence:** HIGH

## Summary

This research investigated how to provision AWS Amplify infrastructure programmatically using Pulumi, enabling Infrastructure-as-Code for complete AWS stack deployment. The current state requires manual Amplify app creation through the AWS Console, which prevents reproducible deployments across environments.

The standard approach uses Pulumi's `@pulumi/aws` provider with `aws.amplify.App` and `aws.amplify.Branch` resources to provision Amplify applications. For GitHub Actions deployment using OIDC authentication (already in use for this project), an IAM OpenID Connect provider must be created to trust GitHub's OIDC provider, along with an IAM role that grants Amplify deployment permissions.

**Primary recommendation:** Use `aws.amplify.App` and `aws.amplify.Branch` resources with `aws.iam.OpenIdConnectProvider` and `aws.iam.Role` for GitHub Actions OIDC authentication. Store Amplify app ID, branch name, and region in SSM Parameter Store for workflow reference.

### Pattern 1: Amplify App Component

**What:** Creates AWS Amplify application with branch configuration and SSM parameter storage.

**When to use:** When provisioning frontend infrastructure for SSR deployment.

**Example:**
```typescript
// Source: Pulumi AWS Registry - https://www.pulumi.com/registry/packages/aws/api-docs/amplify/app/

import * as pulumi from "@pulumi/pulumi";
import * as aws from "@pulumi/aws";

export interface AmplifyFrontendOptions {
  config: InfrastructureConfig;
  backendUrl: pulumi.Input<string>;
  branchName?: string;
  framework?: string;
  stage?: string;
}

export interface AmplifyFrontendOutputs {
  appId: pulumi.Output<string>;
  appArn: pulumi.Output<string>;
  defaultDomain: pulumi.Output<string>;
  branchName: string;
  ssmAppIdPath: string;
  ssmBranchNamePath: string;
  ssmRegionPath: string;
}

export function createAmplifyFrontend(options: AmplifyFrontendOptions): AmplifyFrontendOutputs {
  const branchName = options.branchName || "main";
  const stage = options.stage || (options.config.environment === "prod" ? "PRODUCTION" : "DEVELOPMENT");

  // Create Amplify app
  const app = new aws.amplify.App(`${options.config.environment}-viberator-frontend`, {
    name: `${options.config.environment}-viberator-frontend`,
    description: `Viberator frontend - ${options.config.environment} environment`,
    platform: "WEB_COMPUTE", // Required for SSR
    buildSpec: `version: 1
backend:
  phases:
    preBuild:
      commands:
        - npm ci --legacy-peer-deps
    build:
      commands:
        - npm run build
  artifacts:
    baseDirectory: .next
    files:
      - '**/*'
  cache:
    paths:
      - node_modules/**/*`,
    environmentVariables: {
      NEXT_PUBLIC_API_URL: pulumi.interpolate`${options.backendUrl}`,
    },
    // Disable auto-branch creation for security
    enableAutoBranchCreation: false,
    tags: options.config.tags,
  });

  // Create main branch
  const branch = new aws.amplify.Branch(`${options.config.environment}-viberator-frontend-branch`, {
    appId: app.id,
    branchName: branchName,
    stage: stage,
    framework: options.framework || "Next.js - SSR",
    enableAutoBuild: false, // Build via GitHub Actions
    environmentVariables: {
      NEXT_PUBLIC_API_URL: pulumi.interpolate`${options.backendUrl}`,
    },
    tags: options.config.tags,
  });

  // Store Amplify configuration in SSM for GitHub Actions
  const appIdParam = new aws.ssm.Parameter(`${options.config.environment}-amplify-app-id`, {
    name: `/viberator/${options.config.environment}/amplify/appId`,
    value: app.id,
    type: "String",
    tags: options.config.tags,
  });

  const branchNameParam = new aws.ssm.Parameter(`${options.config.environment}-amplify-branch-name`, {
    name: `/viberator/${options.config.environment}/amplify/branchName`,
    value: branchName,
    type: "String",
    tags: options.config.tags,
  });

  const regionParam = new aws.ssm.Parameter(`${options.config.environment}-amplify-region`, {
    name: `/viberator/${options.config.environment}/amplify/region`,
    value: options.config.awsRegion,
    type: "String",
    tags: options.config.tags,
  });

  return {
    appId: app.id,
    appArn: app.arn,
    defaultDomain: app.defaultDomain,
    branchName: branchName,
    ssmAppIdPath: appIdParam.name,
    ssmBranchNamePath: branchNameParam.name,
    ssmRegionPath: regionParam.name,
  };
}
```

### Pattern 2: OIDC Provider for GitHub Actions

**What:** Creates IAM OpenID Connect provider and role for GitHub Actions to assume when deploying to Amplify.

**When to use:** When using GitHub Actions for Amplify deployment with OIDC authentication (no long-lived credentials).

**Example:**
```typescript
// Source: AWS Fundamentals - https://awsfundamentals.com/blog/github-actions-to-aws
// Adapted from existing project's OIDC pattern for Pulumi deployments

import * as pulumi from "@pulumi/pulumi";
import * as aws from "@pulumi/aws";

export interface AmplifyOidcOptions {
  config: InfrastructureConfig;
  githubRepository: string; // e.g., "username/repo"
  githubOidcProviderArn?: string; // Default: "arn:aws:iam::108740325835:oidc-provider/token.actions.githubusercontent.com"
}

export interface AmplifyOidcOutputs {
  providerArn: pulumi.Output<string>;
  roleArn: pulumi.Output<string>;
  roleName: pulumi.Output<string>;
}

export function createAmplifyOidc(options: AmplifyOidcOptions): AmplifyOidcOutputs {
  const providerArn = options.githubOidcProviderArn || "arn:aws:iam::108740325835:oidc-provider/token.actions.githubusercontent.com";

  // Create IAM OIDC provider for GitHub (skip if already exists for Pulumi)
  const oidcProvider = new aws.iam.OpenIdConnectProvider("github-oidc-amplify", {
    url: "https://token.actions.githubusercontent.com",
    clientIdList: ["sts.amazonaws.com"],
    thumbprintList: ["6938fd4d98bab03faadb97b34396831e3780aea1"],
  });

  // Create IAM role for GitHub Actions to assume
  const role = new aws.iam.Role(`${options.config.environment}-amplify-github-actions-role`, {
    name: `${options.config.environment}-amplify-github-actions-role`,
    description: `Role for GitHub Actions to deploy Amplify frontend - ${options.config.environment}`,
    assumeRolePolicy: pulumi.interpolate`{
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Principal": {
            "Federated": "${oidcProvider.arn}"
          },
          "Action": "sts:AssumeRoleWithWebIdentity",
          "Condition": {
            "StringEquals": {
              "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
            },
            "StringLike": {
              "token.actions.githubusercontent.com:sub": "repo:${options.githubRepository}:*"
            }
          }
        }
      ]
    }`,
    tags: options.config.tags,
  });

  // Attach Amplify deployment permissions policy
  const amplif yDeploymentPolicy = new aws.iam.RolePolicy(`${options.config.environment}-amplify-deployment-policy`, {
    role: role.id,
    policy: pulumi.interpolate`{
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Action": [
            "amplify:StartDeployment",
            "amplify:GetApp",
            "amplify:GetJob",
            "ssm:GetParameter"
          ],
          "Resource": "*"
        }
      ]
    }`,
  });

  return {
    providerArn: oidcProvider.arn,
    roleArn: role.arn,
    roleName: role.name,
  };
}
```

### Pattern 3: Integration with Main Stack

**What:** Wire Amplify component into main infrastructure stack, passing backend URL from existing outputs.

**When to use:** When integrating Amplify infrastructure into existing Pulumi stack.

**Example:**

### Anti-Patterns to Avoid

- **Manual Amplify app creation:** Defeats IaC benefits, makes environments non-reproducible
- **Storing access tokens in Pulumi:** Security risk - use OIDC instead
- **Using repository + accessToken pattern:** Amplify docs show this for OAuth, but better to use GitHub Actions with OIDC for deployment
- **Enabling auto-branch creation:** Security risk - explicit branch configuration required

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Amplify app provisioning | Manual AWS Console creation | `aws.amplify.App` | IaC, reproducible, version controlled |
| GitHub Actions auth | Long-lived AWS access keys | `aws.iam.OpenIdConnectProvider` + Role | No rotation needed, least privilege |
| Amplify deployment scripts | Custom shell scripts | `aws amplify start-deployment` in workflow | Official AWS CLI, battle-tested |
| SSM parameter storage | Hardcoded secrets in workflows | `aws.ssm.Parameter` | Secure, centralized, versioned |

**Key insight:** The existing project already uses OIDC for Pulumi deployments via GitHub Actions. Reuse this pattern for Amplify deployment rather than introducing long-lived credentials.

## Common Pitfalls

### Pitfall 1: Missing Platform Configuration

**What goes wrong:** Amplify app defaults to static hosting, fails to deploy SSR Next.js app.

**Why it happens:** `platform` property defaults to "WEB" (static), must use "WEB_COMPUTE" for SSR.

**How to avoid:** Always set `platform: "WEB_COMPUTE"` when creating Amplify app for Next.js SSR.

**Warning signs:** Build succeeds but deployment shows static files instead of SSR runtime.

### Pitfall 2: Wrong Build Spec Base Directory

**What goes wrong:** Amplify can't find built Next.js files, deployment fails.

**Why it happens:** SSR builds output to `.next` directory, not root or `out` directory.

**How to avoid:** Set `baseDirectory: .next` in build spec artifacts section.

**Warning signs:** Amplify deployment shows "Build failed" with "no such file or directory" errors.

### Pitfall 3: OIDC Provider Already Exists

**What goes wrong:** Pulumi tries to create OIDC provider but AWS returns "AlreadyExistsException".

**Why it happens:** Project already has GitHub OIDC provider for Pulumi deployments.

**How to avoid:** Either reuse existing provider or use `pulumi import` to import into state. Since project already has OIDC for Pulumi, reference existing provider ARN.

**Warning signs:** `pulumi up` fails with provider already exists error.

### Pitfall 4: Missing Amplify Permissions

**What goes wrong:** GitHub Actions workflow fails with "AccessDenied" when triggering Amplify deployment.

**Why it happens:** IAM role lacks Amplify permissions (amplify:StartDeployment, amplify:GetJob, etc.)

**How to avoid:** Attach inline policy with required Amplify actions to OIDC role.

**Warning signs:** Workflow deployment step fails with 403 Forbidden.

### Pitfall 5: Environment Variables Not Passed to Build

**What goes wrong:** Next.js app can't reach backend API, API calls fail.

**Why it happens:** Amplify build doesn't have `NEXT_PUBLIC_API_URL` environment variable set.

**How to avoid:** Configure environment variables on both App resource (global) and Branch resource (branch-specific).

**Warning signs:** Frontend loads but API calls return network errors or connection refused.

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Manual console setup | Pulumi `aws.amplify.App` | 2025+ | IaC for Amplify provisioning |
| Long-lived access keys | OIDC authentication | 2024 | No credential rotation, more secure |

**Deprecated/outdated:**
- **Access tokens in Pulumi state:** Amplify docs show `accessToken` property, but this is deprecated - use OIDC for GitHub Actions instead
- **Auto-branch creation:** Security risk - explicitly define branches instead

## Open Questions

1. **OIDC Provider Reuse**
   - What we know: Project already has GitHub OIDC provider for Pulumi deployments
   - What's unclear: Whether to reuse existing provider or create separate one for Amplify
   - Recommendation: Reuse existing provider to avoid duplicate resources. The existing OIDC provider ARN pattern is `arn:aws:iam::108740325835:oidc-provider/token.actions.githubusercontent.com`

2. **Existing Amplify App Import**
   - What we know: Amplify app was manually created in console
   - What's unclear: How to import existing app into Pulumi state without recreation
   - Recommendation: Use `pulumi import aws:amplify/app:App <name> <app-id>` to import existing app, then update code to match

3. **Build Spec Path Configuration**
   - What we know: Amplify auto-generates buildspec for Next.js SSR if none provided
   - What's unclear: Whether custom buildspec is needed or auto-generation is sufficient
   - Recommendation: Start with auto-generation, add custom buildspec only if needed for specific requirements
