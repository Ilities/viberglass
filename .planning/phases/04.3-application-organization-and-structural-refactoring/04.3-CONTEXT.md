# Phase 4.3: Application organization and structural refactoring - Context

**Gathered:** 2026-01-20
**Status:** Ready for planning

## Phase Boundary

Improve code organization and structure across the application — directory structure, test organization, and package.json consistency. The goal is a maintainable, predictable codebase structure without changing functionality.

## Implementation Decisions

### Directory structure

**Backend (platform/backend/):**
- Organized by domain/route: `tickets/`, `jobs/`, `clankers/`, etc.
- Within each domain: nested subdirectories (`domain/routes/`, `domain/services/`, `domain/models/`)
- Shared code: `src/lib/{utils,types,config}` for global things, each domain has its own `lib/` if needed
- Platform backend: `platform/backend/{workers,invokers,services}` grouped by functionality
- Frontend (`viberator/app/`): leave Next.js app router structure as-is
- Worker code access: monorepo with package builds (tsup) for shared code
- API routes: `app/api/{tickets,jobs,clankers}/route.ts` domain-based folders
- TypeScript path aliases: yes, add `@/` for root, `@domain/` for each domain

**Directory structure target:**
```
platform/backend/
├── src/
│   ├── lib/           # global shared code
│   │   ├── utils/
│   │   ├── types/
│   │   └── config/
│   ├── tickets/       # domain
│   │   ├── routes/
│   │   ├── services/
│   │   ├── models/
│   │   └── lib/       # domain-specific shared
│   ├── jobs/
│   └── clankers/
├── workers/
├── invokers/
└── services/
```

### Test organization

- Test files co-located in same directory as source: `src/tickets/tickets.test.ts`
- Test naming: `.test.ts` suffix
- Shared fixtures/mocks: `__fixtures__/` and `__mocks__/` alongside each domain
- E2E tests: keep existing `e2e-tests/` at root, organize by feature

### Package structure

- Root `package.json` should have all common scripts runnable (dev, build, test, lint, clean for all workspaces)
- Shared dependencies: Claude's discretion (use npm workspaces to dedupe)
- Build tools/output: Claude's discretion based on each package's needs
- Workspace configuration: Claude's discretion based on project structure

### Claude's Discretion

- Shared dependencies management (npm workspaces deduplication vs root-defined)
- Build tool consistency — tsup for backend packages, Next.js for frontend, whatever makes sense for each
- Workspace references configuration (npm workspaces auto-discovery vs explicit list)
- Exact path alias implementation details

## Specific Ideas

- "All workspaces should be buildable from the root"
- "Main commands should be runnable from the root package.json"
