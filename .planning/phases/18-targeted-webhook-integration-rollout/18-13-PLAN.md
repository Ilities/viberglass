# 18-13 Plan: Custom Inbound Multi-Webhook

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 13  
**PR:** PR-13  
**Estimate:** 1.5 days  
**Status:** Planned  
**Depends on:** 18-03, 18-04

## Objective

Deliver full multi-inbound custom webhook support with targeted list/detail configuration management.

## Scope

In scope:
- multiple inbound custom webhook configurations per integration
- per-config endpoint URL and secret lifecycle
- per-config delivery history and retry

Out of scope:
- custom outbound multi-destination (18-14)

## Functional Requirements

1. Users can create, edit, disable, and delete multiple custom inbound configs.
2. Each config has its own endpoint URL and signing secret.
3. Delivery history is viewable by config and includes all relevant statuses.
4. Retry can be triggered for failed deliveries within a selected config.

## Implementation Tasks

1. Ensure backend inbound CRUD supports unrestricted count for custom provider.
2. Ensure delivery records link to custom inbound config id.
3. Build frontend inbound config list/detail panel for custom integration.
4. Keep payload shape guidance explicit and provider-specific.

## Candidate Files

- `apps/platform-backend/src/api/routes/integrations.ts`
- `apps/platform-backend/src/api/routes/webhooks/custom.routes.ts`
- `apps/platform-backend/src/persistence/webhook/WebhookDeliveryDAO.ts`
- `apps/platform-frontend/src/pages/settings/integration-detail/*`

## Test Plan

1. API tests for create/list/update/delete multiple custom inbound configs.
2. API tests for delivery retrieval and retry scoped by config id.
3. E2E for adding multiple custom inbound configs and switching between them.

## Risks and Mitigations

- Risk: endpoint confusion when multiple configs exist.  
  Mitigation: clear names/labels and copy endpoint actions in UI.
- Risk: secret rotation mistakes.  
  Mitigation: explicit regenerate flow and hidden-by-default secret display.

## Acceptance Criteria

1. Custom integration supports multiple inbound webhook configs.
2. Each config has isolated URL/secret and delivery history.
3. UI provides clear list/detail management for each inbound config.
4. Tests validate multi-config lifecycle.
