# 18-03 Plan: Integration Webhook API v2 (Instance-Scoped)

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 03  
**PR:** PR-03  
**Estimate:** 1.5 days  
**Status:** Planned  
**Depends on:** 18-01, 18-02

## Objective

Replace implicit system-level webhook configuration operations with explicit integration-instance-scoped API contracts.

## Problem Statement

Current webhook API usage in frontend and backend relies on resolving first integration instance by system type in multiple places. This is unsafe and prevents targeted configuration when multiple instances exist.

## Scope

In scope:
- instance-scoped webhook config endpoints
- outbound config list/get model updates where needed
- response DTO consistency for inbound/outbound configs and deliveries

Out of scope:
- frontend route updates (18-04)

## API Direction

1. Keep all configuration operations scoped to integration entity id (`/api/integrations/:id/...`).
2. Support full inbound CRUD with explicit config id.
3. Support outbound CRUD with explicit config id or deterministic single-item semantics by provider when applicable.
4. Add config-scoped deliveries endpoint and retry endpoint behavior.

## Implementation Tasks

1. Refactor `apps/platform-backend/src/api/routes/integrations.ts` webhook subroutes.
2. Remove any “first integration by system” assumptions from route-level logic.
3. Ensure request/response payloads include fields needed by targeted UIs (events, active flags, provider ids, secret/token presence flags).
4. Ensure permission and existence checks are strict and deterministic.

## Candidate Files

- `apps/platform-backend/src/api/routes/integrations.ts`
- `apps/platform-backend/src/webhooks/FeedbackService.ts`
- `apps/platform-backend/src/webhooks/WebhookService.ts`
- `packages/types/src/integration.ts` (if new DTO typing is needed)

## Test Plan

1. Route unit tests for all CRUD paths and common failure statuses.
2. Tests for multiple integration instances under same provider.
3. Tests for deliveries scoped by config id and retry endpoint behavior.

## Risks and Mitigations

- Risk: breaking callers that still use old assumptions.  
  Mitigation: clear deprecation response and frontend update in next plan.
- Risk: inconsistent outbound semantics across providers.  
  Mitigation: define provider-specific capabilities explicitly and enforce them in route layer.

## Acceptance Criteria

1. API endpoints are integration-instance-scoped and deterministic.
2. Multiple instances of same provider can be configured independently.
3. Delivery and retry endpoints can target exact webhook configuration.
4. Old behavior paths are either updated or explicitly deprecated.
