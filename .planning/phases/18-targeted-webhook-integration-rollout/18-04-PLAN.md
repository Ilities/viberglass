# 18-04 Plan: Frontend Routing and API Client Refactor (Instance-Scoped)

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 04  
**PR:** PR-04  
**Estimate:** 1.5 days  
**Status:** Planned  
**Depends on:** 18-03

## Objective

Refactor frontend routing and API clients so webhook configuration is bound to explicit integration entity IDs, not integration system IDs.

## Problem Statement

The frontend currently resolves integration entity through `TicketSystem` and often selects the first result. This fails with multiple instances and blocks targeted per-integration configuration pages.

## Scope

In scope:
- route shape and URL parameter updates
- API client contract changes
- state/hook updates in integration detail pages

Out of scope:
- provider-specific UI components (handled in later plans)

## Implementation Tasks

1. Update routes to identify integration detail pages by entity id.
2. Refactor integration cards/list navigation to pass entity id.
3. Replace `getIntegrationEntityBySystem` style logic in client methods.
4. Update `useIntegrationWebhookSettings` to operate on explicit integration id input.
5. Ensure data reload and optimistic updates do not assume a single instance per provider.

## Test Plan

1. Frontend unit/integration tests for navigation and entity-scoped API calls.
2. E2E check: multiple same-provider integrations render and navigate correctly.
3. Regression check for single-instance integration projects.

## Risks and Mitigations

- Risk: deep links from old URLs break.  
  Mitigation: add redirect fallback for old route shape where practical.
- Risk: stale assumptions in hooks/components.  
  Mitigation: comprehensive grep for `TicketSystem`-based webhook calls.

## Acceptance Criteria

1. Settings pages open and operate by integration entity id.
2. No webhook API call path depends on “first integration by system.”
3. UI supports multiple instances of same provider without ambiguity.
4. Updated e2e test path validates route correctness.
