# 18-08 Plan: Jira Inbound

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 08  
**PR:** PR-08  
**Estimate:** 1 day  
**Status:** Planned  
**Depends on:** 18-01, 18-02, 18-03

## Objective

Deliver production-ready Jira inbound webhook handling with provider-specific parsing, signature, and ticket mapping.

## Scope

In scope:
- Jira inbound signature/header handling
- Jira event mapping and payload field extraction
- config matching by Jira identifiers and integration linkage

Out of scope:
- Jira outbound behavior (18-09)
- Jira UI (18-10)

## Provider Behavior

1. Parse Jira webhook event type and normalize event categories used by the platform.
2. Verify signature/header policy according to configured Jira mode.
3. Map issue and comment payloads into internal ticket creation model.
4. Respect auto-execute and inbound allowed events.

## Implementation Tasks

1. Harden Jira payload parser and event normalization.
2. Verify field extraction for issue key, project key, reporter, comment author.
3. Ensure config resolution is based on Jira-specific identifiers.
4. Improve ignored reason messages for unsupported event actions.

## Candidate Files

- `apps/platform-backend/src/webhooks/providers/jira-provider.ts`
- `apps/platform-backend/src/webhooks/WebhookService.ts`
- `apps/platform-backend/src/api/routes/webhooks/jira.routes.ts`

## Test Plan

1. Unit tests for Jira parser mappings and signature behavior.
2. Integration tests for issue-created/comment-created ticket creation paths.
3. Tests for ignored unsupported Jira events.

## Risks and Mitigations

- Risk: Jira Cloud vs self-hosted payload differences.  
  Mitigation: schema-tolerant parsing with explicit required fields.
- Risk: signature assumptions vary by environment.  
  Mitigation: configuration-driven optional/required signature policy.

## Acceptance Criteria

1. Jira inbound webhook creates tickets correctly for supported events.
2. Signature and event validation behavior is deterministic and test-covered.
3. Config resolution does not rely on GitHub-style metadata.
4. Delivery logs include meaningful Jira event metadata.
