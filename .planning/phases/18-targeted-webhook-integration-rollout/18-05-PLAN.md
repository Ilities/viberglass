# 18-05 Plan: GitHub Inbound

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 05  
**PR:** PR-05  
**Estimate:** 1 day  
**Status:** Planned  
**Depends on:** 18-01, 18-02, 18-03

## Objective

Complete and harden GitHub inbound webhook handling as the first provider-specific production path.

## Scope

In scope:
- GitHub inbound parse/verify/filter behavior
- event handling for `issues` and `issue_comment`
- proper mapping to ticket creation and optional auto-execution

Out of scope:
- GitHub outbound posting (18-06)
- GitHub targeted UI (18-07)

## Provider Behavior

1. Verify `X-Hub-Signature-256` using raw body and configured secret.
2. Deduplicate by `X-GitHub-Delivery`.
3. Parse and filter supported inbound events.
4. Resolve webhook config by integration/config metadata deterministically.
5. Create tickets with rich metadata and optional auto-run trigger.

## Implementation Tasks

1. Finalize GitHub parser and metadata extraction logic.
2. Ensure allowed event filtering is exact and test-covered.
3. Ensure integration/project linkage resolution is deterministic.
4. Improve ignored/rejected diagnostics in delivery records.

## Candidate Files

- `apps/platform-backend/src/webhooks/providers/github-provider.ts`
- `apps/platform-backend/src/webhooks/WebhookService.ts`

- `apps/platform-backend/src/webhooks/deduplication.ts`

## Test Plan

1. Unit tests for valid signature, invalid signature, duplicate deliveries.
2. Event tests for `issues.opened` and bot-triggered comment flows.
3. Integration tests for ticket creation and optional job submission.

## Risks and Mitigations

- Risk: payload variants from GitHub App installations.  
  Mitigation: tolerant parsing with strict required-field checks.
- Risk: accidental event over-processing.  
  Mitigation: explicit event allowlist and action checks.

## Acceptance Criteria

1. GitHub inbound route is stable and deterministic.
2. Duplicate delivery handling is idempotent.
3. Tickets created contain expected external identifiers and metadata.
4. Tests validate critical inbound cases and failure handling.
