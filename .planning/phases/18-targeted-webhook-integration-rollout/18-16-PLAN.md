# 18-16 Plan: Delivery/Retry UX and Observability

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 16  
**PR:** PR-16  
**Estimate:** 1 day  
**Status:** Planned  
**Depends on:** 18-05, 18-06, 18-08, 18-09, 18-11, 18-14, 18-15

## Objective

Unify delivery visibility and retry mechanics across providers with config-scoped observability.

## Scope

In scope:
- backend delivery query contracts and retry behavior normalization
- UI delivery tables/actions across all provider-specific pages
- operational visibility for recent failures and retries

Out of scope:
- provider logic changes unrelated to observability

## Implementation Tasks

1. Standardize delivery API shape for inbound/outbound deliveries.
2. Ensure queries are config-scoped and include all statuses as needed.
3. Normalize retry endpoint behavior and response semantics.
4. Update provider pages to show consistent delivery status rows and actions.
5. Add minimal telemetry/logging conventions for retry attempts.

## Candidate Files

- `apps/platform-backend/src/api/routes/integrations.ts`
- `apps/platform-backend/src/persistence/webhook/WebhookDeliveryDAO.ts`
- `apps/platform-backend/src/webhooks/deduplication.ts`
- `apps/platform-frontend/src/pages/settings/integration-detail/*`

## Test Plan

1. API tests for delivery list filtering and retry behavior.
2. UI tests for status rendering and retry actions.
3. E2E for failed-delivery retry and state refresh.

## Risks and Mitigations

- Risk: retry semantics diverge between providers.  
  Mitigation: central retry contract with provider-specific adapters.
- Risk: high-volume delivery history affects performance.  
  Mitigation: pagination and capped fetch defaults.

## Acceptance Criteria

1. Delivery history is config-scoped and provider-consistent.
2. Retry flow works from UI to backend and returns clear state changes.
3. Operational diagnostics are sufficient for support and debugging.
4. Automated tests cover list/retry core paths.
