# 18-02 Plan: DB and DAO Corrections for Multi-Config and Provider Coverage

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 02  
**PR:** PR-02  
**Estimate:** 1 day  
**Status:** Planned  
**Depends on:** 18-01

## Objective

Correct schema and persistence gaps to support multi-config behavior and proper provider coverage, including delivery tracking by webhook configuration.

## Problem Statement

The current data model has functional gaps:
- provider constraints are not consistently aligned with actual providers in use
- delivery records are not config-scoped, limiting observability and retry targeting
- DAO query helpers are not optimized for config-specific delivery views

## Scope

In scope:
- migration updates
- DAO query additions and updates
- indexing for config-scoped delivery history

Out of scope:
- frontend UI changes
- endpoint contract overhaul (handled in 18-03)

## Implementation Tasks

1. Add migration to ensure provider constraints include `shortcut` where required.
2. Add `webhook_config_id` to delivery attempts table.
3. Add indexes for `webhook_config_id`, status, created timestamp.
4. Update `WebhookDeliveryDAO` methods for config-scoped list/retry queries.
5. Update `WebhookConfigDAO` helpers where needed to support list/get patterns used by API v2.

## Candidate Files

- `apps/platform-backend/src/migrations/*`
- `apps/platform-backend/src/persistence/webhook/WebhookConfigDAO.ts`
- `apps/platform-backend/src/persistence/webhook/WebhookDeliveryDAO.ts`
- `apps/platform-backend/src/persistence/types/database.ts`

## Test Plan

1. Migration test against empty and existing schema states.
2. DAO unit tests for config-scoped delivery listing and lookup.
3. Verify existing webhook flows still persist and deduplicate correctly.

## Risks and Mitigations

- Risk: migration compatibility issues in existing environments.  
  Mitigation: additive migration strategy, backfill script, rollback path.
- Risk: query regressions on delivery history views.  
  Mitigation: add DAO tests with mixed provider/config fixtures.

## Acceptance Criteria

1. Schema supports provider set: GitHub/Jira/Shortcut/Custom.
2. Delivery rows can be filtered reliably by `webhook_config_id`.
3. DAO APIs expose config-scoped query paths needed by new endpoints.
4. Migration is reversible and documented.
