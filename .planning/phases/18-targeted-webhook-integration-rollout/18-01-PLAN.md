# 18-01 Plan: Webhook Core Stabilization

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 01  
**PR:** PR-01  
**Estimate:** 1.5 days  
**Status:** Planned  
**Depends on:** none

## Objective

Stabilize webhook processing foundations so provider-specific work is built on a correct routing, signature, and config-resolution base.

## Problem Statement

Current webhook flow mixes route-level assumptions and service-level assumptions:
- provider resolution and config lookup are not consistently provider-aware
- signature handling differs by route and service behavior
- custom webhook signature verification currently risks raw-body mismatch behavior

These issues make provider-specific development fragile.

## Scope

In scope:
- inbound webhook processing reliability for GitHub/Jira/Shortcut/custom
- signature verification correctness and consistency
- provider-aware configuration resolution in `WebhookService`

Out of scope:
- new UI features
- outbound multi-destination design

## Implementation Tasks

1. Refactor provider-aware config resolution in `apps/platform-backend/src/webhooks/WebhookService.ts`.
2. Normalize signature header extraction and verification flow per provider.
3. Remove conflicting assumptions between route middleware and service methods.
4. Fix custom inbound signature verification to use raw bytes (not reserialized body).
5. Ensure rejected/ignored/failed status handling remains explicit and observable.

## Test Plan

1. Add/expand unit tests for provider-specific signature failures/successes.
2. Add tests for config resolution across GitHub/Jira/Shortcut/custom metadata paths.
3. Add tests for duplicate delivery handling and non-regression on status response codes.
4. Run backend unit suite.

## Risks and Mitigations

- Risk: changing signature flow breaks existing integrations.  
  Mitigation: preserve header contract and add route-level tests per provider.
- Risk: config resolution changes can misroute events.  
  Mitigation: explicit provider + providerProjectId + integrationId matching tests.

## Acceptance Criteria

1. All provider routes process valid payloads and reject invalid signatures consistently.
2. Webhook service resolves the correct active config without provider bias.
3. Custom webhook verification uses exact raw body bytes.
4. Unit tests cover provider routing, signature, dedup, and result statuses.
