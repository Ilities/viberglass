# 18-15 Plan: Custom Outbound Targeted UI

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 15  
**PR:** PR-15  
**Estimate:** 1.5 days  
**Status:** Planned  
**Depends on:** 18-04, 18-14

## Objective

Create a dedicated custom outbound destination management UI with list/detail editing and test-send support.

## Scope

In scope:
- custom outbound destinations list
- create/edit/delete destination forms
- event selection and auth/header controls
- test-send action

Out of scope:
- backend transport logic changes (already in 18-14)

## UI Behavior

1. Destination list with status badges and event summary.
2. Detail editor for URL/method/headers/auth/retry settings.
3. Event toggles for `job_started` and `job_ended`.
4. Test-send action for selected destination with response summary.

## Implementation Tasks

1. Build custom outbound destination list and editor components.
2. Wire API calls for target CRUD operations.
3. Add client-side validation for URL/method/header schemas.
4. Add test-send invocation and result notification UX.
5. Ensure masked display for sensitive fields.

## Candidate Files

- `apps/platform-frontend/src/pages/settings/integration-detail/*`
- `apps/platform-frontend/src/service/api/integration-api.ts`
- `apps/platform-frontend/src/components/*` (new destination form/list components)

## Test Plan

1. Component tests for validation, save, and delete flows.
2. E2E for multi-destination create/edit/delete.
3. E2E for event toggles and test-send action.

## Risks and Mitigations

- Risk: complex form leads to usability regressions.  
  Mitigation: progressive disclosure and clear defaults.
- Risk: accidental secret exposure in UI.  
  Mitigation: masked values, explicit reveal controls, and no plaintext echo.

## Acceptance Criteria

1. Custom outbound UI is destination-oriented and non-generic.
2. Users can manage multiple destinations end-to-end.
3. Sensitive fields are handled safely in view/edit modes.
4. Test-send flow is available and reports clear result status.
