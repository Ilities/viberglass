# 18-11 Plan: Shortcut Inbound and Outbound Core

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 11  
**PR:** PR-11  
**Estimate:** 1.5 days  
**Status:** Planned  
**Depends on:** 18-01, 18-02, 18-03

## Objective

Implement stable provider-specific inbound and outbound behavior for Shortcut.

## Scope

In scope:
- Shortcut inbound signature and event parsing
- Shortcut outbound comment/label/state updates
- target resolution and event toggle behavior

Out of scope:
- Shortcut targeted UI (18-12)

## Provider Behavior

Inbound:
1. Verify signature from Shortcut headers using configured secret.
2. Parse story/comment events and map to internal ticket event types.
3. Enforce event allowlist and deduplication.

Outbound:
1. Post job lifecycle comments to stories.
2. Apply configured label/state updates on completion/failure.
3. Respect outbound event toggles.

## Implementation Tasks

1. Harden `shortcut-provider` parser and signature paths.
2. Ensure webhook config resolution can use Shortcut identifiers.
3. Implement outbound formatting and API call stability checks.
4. Update feedback resolution logic for Shortcut ticket metadata.

## Candidate Files

- `apps/platform-backend/src/webhooks/providers/shortcut-provider.ts`
- `apps/platform-backend/src/webhooks/WebhookService.ts`

- `apps/platform-backend/src/api/routes/webhooks/shortcut.routes.ts`

## Test Plan

1. Unit tests for inbound parse/verify/dedup.
2. Unit tests for outbound comment/label/state operations.
3. Integration tests for ticket creation and outbound feedback dispatch.

## Risks and Mitigations

- Risk: payload shape variance over Shortcut event versions.  
  Mitigation: robust optional parsing with required-field guards.
- Risk: outbound state update assumptions differ by workspace.  
  Mitigation: keep state updates configurable and optional.

## Acceptance Criteria

1. Shortcut inbound reliably processes supported events.
2. Shortcut outbound posts expected lifecycle updates.
3. Event toggle and config matching behavior is test-covered.
4. Delivery records include enough metadata for debugging.
