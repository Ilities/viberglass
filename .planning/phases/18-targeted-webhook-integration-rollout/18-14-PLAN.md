# 18-14 Plan: Custom Outbound Multi-Destination

**Phase:** 18-targeted-webhook-integration-rollout  
**Plan:** 14  
**PR:** PR-14  
**Estimate:** 2 days  
**Status:** Planned  
**Depends on:** 18-13

## Objective

Add outbound support for custom provider with multiple independently-configurable destinations.

## Scope

In scope:
- multiple outbound custom targets per integration
- per-target URL/method/headers/auth and event filters
- retry strategy and delivery tracking integration

Out of scope:
- custom outbound targeted UI implementation details (18-15)

## Data and Contract Direction

Each outbound target should support:
- name/label
- target URL
- HTTP method
- headers/auth material
- enabled events (`job_started`, `job_ended`)
- active flag
- optional signing secret and signature algorithm
- retry policy fields (attempt count, backoff)

## Implementation Tasks

1. Add migration(s) for custom outbound target storage model.
2. Implement backend CRUD endpoints for custom outbound targets.
3. Extend `FeedbackService` dispatch to fan-out to matching active custom targets.
4. Add per-target delivery tracking and failure diagnostics.
5. Ensure one failing destination does not block others.

## Candidate Files

- `apps/platform-backend/src/migrations/*`
- `apps/platform-backend/src/api/routes/integrations.ts`

- `apps/platform-backend/src/webhooks/providers/custom-provider.ts`
- `apps/platform-backend/src/persistence/webhook/*`

## Test Plan

1. Migration and DAO tests for target CRUD.
2. Dispatch tests for multi-target fan-out behavior.
3. Failure-isolation tests (one target fails, others succeed).
4. Event filter tests for `job_started`/`job_ended` combinations.

## Risks and Mitigations

- Risk: outbound fan-out introduces latency spikes.  
  Mitigation: bounded concurrency + timeout controls.
- Risk: sensitive header material handling errors.  
  Mitigation: secret storage flags, redaction, and strict serialization.

## Acceptance Criteria

1. Custom integration supports multiple outbound destinations.
2. Each destination can be independently configured and toggled.
3. Dispatch fan-out works with failure isolation and logging.
4. Tests validate event filtering and retry behavior.
