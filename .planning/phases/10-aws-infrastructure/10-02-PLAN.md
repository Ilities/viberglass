<task type="auto">
  <name>Task 1: Create VPC with public and private subnets</name>
  <files>infrastructure/components/vpc.ts</files>
  <action>
Create `infrastructure/components/vpc.ts` with VPC networking:

1. Create VPC:
   - Configurable CIDR block (default 10.0.0.0/16)
   - Enable DNS hostnames and support
   - Tags with environment and name

2. Create Internet Gateway:
   - Attached to VPC
   - For public subnet internet access

3. Create Public Subnets:
   - 2 subnets in different AZs (eu-west-1a, eu-west-1b)
   - CIDR: 10.0.1.0/24, 10.0.2.0/24
   - Map public IP on launch
   - Route table with 0.0.0.0/0 to Internet Gateway

4. Create Private Subnets:
   - 2 subnets in different AZs for high availability
   - CIDR: 10.0.10.0/24, 10.0.11.0/24
   - No public IP on launch
   - Route table to NAT gateway (created in Task 2)

5. Create Elastic IPs and NAT Gateways:
   - One NAT gateway per AZ for reliability
   - In public subnets
   - Private subnet route to NAT for internet access

6. Export:
   - vpcId
   - publicSubnetIds (array)
   - privateSubnetIds (array)
   - natGatewayIds (array)

Use `awsx.ec2.Vpc` if available for simpler setup, otherwise build from individual `aws.ec2.*` resources.
  </action>
  <verify>
    - `cd infrastructure && npx tsc` compiles without errors
    - `pulumi preview --stack dev` shows VPC, subnets, IGW, NAT gateways
    - Exported values include vpcId and subnet arrays
  </verify>
  <done>
    VPC component creates public/private subnets across 2 AZs. NAT gateways provide private subnet egress. All resources are properly tagged.
  </done>
</task>

<task type="auto">
  <name>Task 2: Create security groups for application layers</name>
  <files>infrastructure/components/vpc.ts</files>
  <action>
Add security groups to `infrastructure/components/vpc.ts`:

1. Create Backend Security Group:
   - Allows inbound HTTP (80) from load balancer/security group
   - Allows inbound HTTPS (443) from load balancer/security group
   - Allows inbound custom port (default 3000) from VPC CIDR for internal communication
   - Allows all outbound traffic

2. Create RDS Security Group:
   - Allows inbound PostgreSQL (5432) ONLY from backend security group
   - No inbound from internet
   - Allows all outbound traffic

3. Create Worker Security Group:
   - Allows inbound from backend/security group for callbacks
   - Allows all outbound traffic (for git, API calls)

4. Export security group IDs:
   - backendSecurityGroupId
   - rdsSecurityGroupId
   - workerSecurityGroupId

Use `aws.ec2.SecurityGroup` with ingress/egress rules. Reference VPC ID from earlier in the same file.
  </action>
  <verify>
    - `pulumi preview --stack dev` shows 3 security groups
    - Backend SG allows inbound on ports 80, 443, 3000
    - RDS SG allows inbound on 5432 only from backend SG
    - All SGs allow outbound traffic
  </verify>
  <done>
    Three security groups created with appropriate ingress rules. RDS is isolated from the internet. Backend can communicate with RDS and workers.
  </done>
</task>
