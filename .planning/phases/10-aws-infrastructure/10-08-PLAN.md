<task type="auto">
  <name>Task 1: Create S3 bucket for frontend static hosting</name>
  <files>infrastructure/components/frontend.ts</files>
<action>
Create `infrastructure/components/frontend.ts` with frontend infrastructure:

1. Create S3 Bucket for frontend:
   - Use `aws.s3.BucketV2`
   - Name: Configurable prefix + environment + random suffix
   - Different from uploads bucket
   - Example: `viberator-${environment}-frontend-{random}`

2. Configure bucket for static website hosting:
   - Use `aws.s3.BucketWebsiteConfigurationV2`
   - Index document: index.html
   - Error document: 404.html (or index.html for SPA routing)

3. Enable bucket encryption:
   - Use `aws.s3.BucketServerSideEncryptionConfiguration`
   - SSE algorithm: AES256

4. Block public access:
   - BlockPublicPolicy: false (needed for static site)
   - BlockPublicAcls: false
   - Other settings: true

5. Create bucket policy for public read:
   - Use `aws.s3.BucketPolicy`
   - Allow s3:GetObject from all principals
   - Resource: bucket ARN + /*

6. Export bucket name and website endpoint

Note: For CloudFront, the bucket doesn't need public access. The policy below is for static-only without CloudFront.
  </action>
  <verify>
    - `cd infrastructure && npx tsc` compiles without errors
    - `pulumi preview --stack dev` shows frontend S3 bucket
    - Bucket has website configuration with index.html
  </verify>
  <done>
    S3 bucket created for frontend hosting. Website configuration enabled. Public read policy for static access (or CloudFront OAI if used).
  </done>
</task>

<task type="auto">
  <name>Task 2: Create CloudFront distribution</name>
  <files>infrastructure/components/frontend.ts</files>
<action>
Add CloudFront distribution to `infrastructure/components/frontend.ts`:

1. Create CloudFront Distribution:
   - Use `aws.cloudfront.Distribution`
   - Enabled: true
   - Price class: PriceClass_100 for dev/staging, PriceClass_All for prod
   - Origins:
     - S3 origin with custom origin config or S3 origin
     - Use OAI (Origin Access Identity) or OAC for security
   - Default cache behavior:
     - Target origin ID: S3 origin
     - Viewer protocol policy: redirect-to-https
     - Allowed methods: GET, HEAD, OPTIONS
     - Cached methods: GET, HEAD
     - Compress: true
     - TTL: Default 86400, min 0, max 31536000
   - Custom error responses:
     - 403/404: Response code 200, /index.html (for SPA routing)
   - Default root object: index.html
   - Custom error responses for Next.js SPA routing:
     - ErrorCode: 403, ResponseCode: 200, ResponsePagePath: /index.html
     - ErrorCode: 404, ResponseCode: 200, ResponsePagePath: /index.html

2. Create Origin Access Control (OAC):
   - Use `aws.cloudfront.OriginAccessControl`
   - Name: `viberator-${environment}-frontend-oac`
   - Origin access control type: s3

3. Update S3 bucket policy to allow CloudFront OAC:
   - Remove public read
   - Add policy allowing cloudfront.amazonaws.com with OAC

4. Export:
   - cloudfrontDomainName
   - cloudfrontDistributionId

Use `aws.cloudfront.Distribution` with originAccessControlId.
  </action>
  <verify>
    - `pulumi preview --stack dev` shows CloudFront distribution
    - OAC created and S3 policy updated
    - Custom error responses for SPA routing
  </verify>
  <done>
    CloudFront distribution created with OAC. S3 bucket not publicly accessible. SPA routing handled via 403/404 redirects. HTTPS enforced.
  </done>
</task>
