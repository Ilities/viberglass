<task type="auto">
  <name>Task 1: Create Application Load Balancer</name>
  <files>infrastructure/components/load-balancer.ts</files>
  <action>
Create `infrastructure/components/load-balancer.ts` with ALB infrastructure:

1. Create Application Load Balancer:
   - Use `aws.lb.LoadBalancer`
   - Name: `viberator-${environment}-alb`
   - Internal: false (internet-facing)
   - Subnets: VPC public subnets (2 for HA)
   - Security group: Allows HTTP/HTTPS from internet, forwards to backend SG
   - Tags: Environment, Application

2. Create ALB Security Group:
   - Inbound 80 from 0.0.0.0/0
   - Inbound 443 from 0.0.0.0/0
   - Outbound to backend security group
   - Use `aws.ec2.SecurityGroup`

3. Create Target Group:
   - Use `aws.lb.TargetGroup`
   - Name: `viberator-${environment}-backend-tg`
   - Port: 80 (container port, assumes SSL termination at ALB)
   - Protocol: HTTP
   - VPC ID: from VPC component
   - Health check:
     - Path: /health
     - Interval: 30 seconds
     - Timeout: 5 seconds
     - Healthy threshold: 3
     - Unhealthy threshold: 3
     - Matcher: HTTP 200

4. Create HTTP Listener (port 80):
   - Use `aws.lb.Listener`
   - Default action: Redirect to HTTPS (301)

5. Create HTTPS Listener (port 443):
   - Use `aws.lb.Listener`
   - Default action: Forward to target group
   - Certificate: ACM certificate ARN from stack config (optional, use HTTP-only for MVP)

6. Export:
   - albDnsName
   - albCanonicalHostedZoneId
   - targetGroupArn
   - albSecurityGroupId

For MVP, HTTP-only is acceptable. Add HTTPS with ACM certificate in production.
  </action>
  <verify>
    - `cd infrastructure && npx tsc` compiles without errors
    - `pulumi preview --stack dev` shows ALB, target group, listeners
    - ALB is in public subnets, target group references VPC
  </verify>
  <done>
    ALB created in public subnets. Target group configured for health checks. HTTP redirects to HTTPS (or HTTP-only for MVP). ALB DNS name exported.
  </done>
</task>

<task type="auto">
  <name>Task 2: Create backend ECS task definition</name>
  <files>infrastructure/components/backend-ecs.ts</files>
  <action>
Create `infrastructure/components/backend-ecs.ts` with backend ECS infrastructure:

1. Create ECR image reference:
   - Use `awsx.ecr.Image` or reference existing ECR repository
   - Build from `platform/backend/`
   - Dockerfile path: `platform/backend/Dockerfile` (to be created or use existing)
   - Platform: linux/amd64

2. Create IAM role for backend task execution:
   - Use `aws.iam.Role`
   - Assume role policy: ecs-tasks.amazonaws.com
   - Attach AmazonECSTaskExecutionRolePolicy

3. Create IAM role for backend task:
   - Assume role policy: ecs-tasks.amazonaws.com
   - Inline policies for:
     - SSM:GetParameter (for database URL and secrets)
     - S3:GetObject/PutObject (for file uploads)
     - ECS:ExecuteCommand (for AWS ECS Exec)
     - KMS:Decrypt (for SSM parameters)
     - CloudWatch Logs (for logging)

4. Create backend task definition:
   - Use `aws.ecs.TaskDefinition`
   - Family: `viberator-${environment}-backend`
   - Network mode: awsvpc
   - Requires compatibilities: FARGATE
   - CPU: Configurable (default: 256 for dev, 512 for prod)
   - Memory: Configurable (default: 512 for dev, 1024 for prod)
   - Execution role ARN: from execution role
   - Task role ARN: from task role
   - Container definition:
     - Name: viberator-backend
     - Image: from ECR
     - Essential: true
     - Port mappings: 3000 (app port)
     - Environment variables:
       - NODE_ENV: production
       - PORT: 3000
       - AWS_REGION: from config
     - Secrets:
       - DATABASE_URL: From SSM parameter
       - Any other sensitive config
     - Log configuration: From logging component

5. Export task definition ARN and family

Use `aws.ecs.TaskDefinition` with containerDefinitions as JSON.stringify.
  </action>
  <verify>
    - `pulumi preview --stack dev` shows backend task definition
    - Task has container with port 3000
    - Secrets reference SSM parameters
    - IAM role has SSM, S3, KMS permissions
  </verify>
  <done>
    Backend task definition created with container configuration. Database URL from SSM secrets. IAM role has least-privilege permissions.
  </done>

   - Desired count: Configurable (1 for dev, 2 for prod)
   - Launch type: FARGATE
   - Force new deployment: true
   - Network configuration:
     - Subnets: VPC private subnets
     - Security groups: [backend security group, ALB security group]
   - Load balancers:
     - Target group ARN: from load balancer component
     - Container name: viberator-backend
     - Container port: 3000

2. Create auto-scaling target:
   - Use `aws.appautoscaling.Target`
   - Resource: ECS service
   - Scalable dimension: ecs:service:DesiredCount
   - Service namespace: ecs
   - Min capacity: 1 (dev) or 2 (prod)
   - Max capacity: 3 (dev) or 10 (prod)

3. Create auto-scaling policies:
   - CPU-based: Scale out at 70%, scale in at 30%
   - Memory-based: Scale out at 80%, scale in at 40%
   - Use `aws.appautoscaling.Policy`

4. Export:
   - backendServiceName
   - backendServiceArn
   - backendScalingTargetArn

Use `aws.ecs.Service` with desiredCount and networkConfiguration.
