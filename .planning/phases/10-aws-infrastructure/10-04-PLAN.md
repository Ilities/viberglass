<task type="auto">
  <name>Task 1: Create S3 bucket with encryption</name>
  <files>infrastructure/components/storage.ts</files>
  <action>
Create `infrastructure/components/storage.ts` with S3 infrastructure:

1. Create S3 Bucket:
   - Name: Configurable prefix + environment + random suffix
   - Region: Same as AWS region config
   - Tags: Environment, Application, Managed-by

2. Enable bucket encryption:
   - Use `aws.s3.BucketServerSideEncryptionConfiguration`
   - SSE algorithm: AES256 (aws:kms also available if needed)
   - Bucket key enabled for cost optimization

3. Block public access:
   - BlockPublicPolicy: true
   - BlockPublicAcls: true
   - IgnorePublicAcls: true
   - RestrictPublicBuckets: true

4. Enable versioning:
   - Use `aws.s3.BucketVersioning`
   - Enabled for production (to recover from accidental deletion)
   - Suspended for dev (to save cost)

5. Create bucket for each environment:
   - dev: viberator-dev-uploads-{random}
   - staging: viberator-staging-uploads-{random}
   - prod: viberator-uploads-{random}

6. Export bucket name and ARN

Use `aws.s3.BucketV2` for newer resource type.
  </action>
  <verify>
    - `cd infrastructure && npx tsc` compiles without errors
    - `pulumi preview --stack dev` shows S3 bucket with encryption
    - Block public access settings are enabled
  </verify>
  <done>
    S3 bucket created with server-side encryption. Public access is blocked. Versioning configured appropriately per environment.
  </done>
</task>

<task type="auto">
  <name>Task 2: Add lifecycle policies for cost optimization</name>
  <files>infrastructure/components/storage.ts</files>
  <action>
Add lifecycle configuration to `infrastructure/components/storage.ts`:

1. Create lifecycle rule:
   - Use `aws.s3.BucketLifecycleConfigurationV2`
   - Rule name: "delete-old-objects"

2. For dev environment:
   - Abort incomplete multipart uploads after 7 days
   - Delete noncurrent versions after 7 days
   - Expire current objects after 90 days (dev cleanup)

3. For staging:
   - Abort incomplete multipart uploads after 7 days
   - Transition noncurrent versions to IA after 30 days
   - Delete noncurrent versions after 90 days
   - No expiration on current objects

4. For prod:
   - Abort incomplete multipart uploads after 7 days
   - Transition current objects to IA after 30 days
   - Transition current objects to Glacier after 90 days
   - Transition current objects to Deep Archive after 180 days
   - Noncurrent versions to Glacier after 30 days
   - Delete noncurrent versions after 365 days

5. Use environment variable to select appropriate lifecycle rules

Use `aws.s3.BucketLifecycleConfigurationV2` with rules array.
  </action>
  <verify>
    - `pulumi preview --stack dev` shows lifecycle rule with 90-day expiration
    - `pulumi preview --stack prod` shows transition rules (IA -> Glacier -> Deep Archive)
    - Multipart upload cleanup configured for all environments
  </verify>
  <done>
    Lifecycle policies optimize storage costs. Dev has aggressive cleanup. Prod has tiered storage archiving. Multipart uploads don't linger.
  </done>
</task>

<task type="auto">
  <name>Task 3: Create IAM policy for S3 access</name>
  <files>infrastructure/components/storage.ts</files>
<action>
Add IAM policy to `infrastructure/components/storage.ts`:

1. Create IAM policy for S3 access:
   - Use `aws.iam.Policy`
   - Name: `${environment}-viberator-s3-access-policy`
   - Policy statements:
     a) Allow PutObject, GetObject, DeleteObject on bucket/*
     b) Allow ListBucket on bucket
     c) Allow GetObject on objects with Deny if not via presigned URL (optional)

2. Policy JSON structure:
```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:GetObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::${bucket}/*"
    },
    {
      "Effect": "Allow",
      "Action": ["s3:ListBucket"],
      "Resource": "arn:aws:s3:::${bucket}"
    }
  ]
}
```

3. Export policy ARN for attachment to ECS task roles / Lambda roles

This policy will be attached to backend ECS task role and worker roles.

Use `aws.iam.Policy` with template string for bucket ARN.
  </action>
  <verify>
    - `pulumi preview --stack dev` shows IAM policy for S3 access
    - Policy allows PutObject, GetObject, DeleteObject, ListBucket
    - Policy ARN is exported
  </verify>
  <done>
    IAM policy created for S3 bucket access. Policy ARN exported for attachment to compute roles. Permissions scoped to specific bucket only.
  </done>
